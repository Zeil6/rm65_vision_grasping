# 技术细节与实现说明

> 本文档根据毕业设计终稿与仓库公开代码交叉整理。目标不是把论文整段搬过来，而是回答工程面试中最常被追问的问题：系统数据怎么流动、坐标怎么算、MoveIt 如何执行、参数为什么这样设，以及哪些功能仍有边界。

## 1. 任务定义与实现范围

系统要完成的闭环是：

```text
RGB-D 图像
  → 颜色/形状目标识别
  → 像素与深度恢复三维点
  → 相机坐标转换到机械臂基坐标
  → 生成预抓取与抓取位姿
  → IK 求解与关节轨迹执行
  → 灵巧手闭合、提起与释放
```

实验对象按“三种颜色 × 两种形状”划分为 6 个类别。形状参与抓取策略选择：圆形目标对应包络抓取，方形目标对应捏取抓取；颜色主要用于类别区分，不改变抓取方式。

### 完成度说明

| 模块 | 论文实验 | 当前公开代码 |
|---|---|---|
| 自建数据集与 YOLO 训练 | 已完成 | 未发布原始数据与 120+ MB 权重 |
| YOLO 3D 检测接入 | 已联调 | 通过 `/yolo/detections_3d` 接入 |
| Eye-in-Hand 标定 | 使用 RealMan 官方工具完成 | 发布外参快照和 TF 接入方式 |
| RM65 IK 与轨迹执行 | 已联调 | 提供完整 ROS 2 节点 |
| 灵巧手基础开合与规则映射 | 论文实验中完成基础动作验证 | 驱动相关实现未公开，函数保留为接口占位 |
| 高级多指力控/触觉闭环 | 未实现 | 未实现 |

这个区分很重要：仓库展示的是可审阅的系统集成核心，不把第三方 SDK、模型权重或未公开的硬件控制接口伪装成自己的代码。

## 2. 硬件选型与机器人学含义

| 硬件 | 论文记录的关键参数 | 在系统中的作用 |
|---|---|---|
| RealMan RM65-6F | 6 DoF；额定负载 5 kg；工作半径 638.5 mm；重复定位精度 ±0.05 mm | 负责目标接近、抓取位姿调整与搬运 |
| ROH-LiteS001 | 11 个总自由度、6 个主动自由度；约 540 g；重复定位精度 ±1 mm | 执行五指张开、包络或捏取动作 |
| Intel RealSense D435i | 主动双目；典型 0.3-3 m；深度最高 1280×720@30 fps；约 87°×58° 深度视场 | 同时提供彩色图像、深度和相机内参 |

机械臂的 5 kg 额定负载并不等于末端可以随意加设备。灵巧手、相机、支架和线缆共同改变末端负载、质心和转动惯量，因此真机部署需要：

1. 向控制器配置正确的工具质量与质心；
2. 使用较低速度/加速度完成首轮联调；
3. 避免线缆牵拉改变末端运动或标定外参；
4. 相机或支架重新安装后重新标定。

## 3. 感知模块：YOLO 为什么比 HSV 阈值更适合

### 3.1 数据集构建

数据来自实际抓取环境，并围绕 6 个颜色-形状组合类别人工标注。采集时重点覆盖：

- 不同摆放位置和拍摄角度；
- 不同背景与光照；
- 部分遮挡和目标不完全进入画面；
- 与真实抓取时相近的相机高度和工作区域。

训练采用预训练权重继续微调，而不是从零训练。这样能利用通用视觉特征，降低小规模自建数据集训练的不稳定性。

论文保留了训练曲线和检测示例，但没有完整记录 `epochs`、`batch`、`imgsz`、数据集划分数量及最终数值指标。因此，本仓库不编造 mAP 或成功率；若要做严格复现，应从原训练目录的 `args.yaml`、`results.csv` 和数据配置文件恢复这些参数。

### 3.2 推理输出

识别模块输出：

- 类别标签 `class_id/class_name`；
- 置信度 `confidence`；
- 二维检测框；
- 与深度图结合得到的三维包围框中心。

当前坐标转换节点订阅：

```text
/yolo/detections_3d  (yolo_msgs/DetectionArray)
```

并读取第一项检测结果的 `bbox3d.center.position`。生产级系统还应加入目标类别白名单、置信度阈值、最近目标选择或跟踪 ID，避免“检测列表第一项”在多目标场景中跳变。

### 3.3 OpenCV 对照基线

传统方案流程为：

```text
BGR → HSV → 颜色阈值掩膜 → 轮廓筛选 → 多边形近似 → 颜色/形状类别
```

它在背景干净、光照稳定时直观且容易调试，但存在两个结构性弱点：颜色依赖人工阈值，形状依赖二值轮廓完整性。论文实验中，白色圆球接近浅白背景时出现漏检；而 YOLO 在背景干扰、角度变化和部分目标离开画面时仍能给出更稳定结果。因此最终将 OpenCV 保留为可解释基线，正式感知入口采用 YOLO。

## 4. 从检测框到相机三维点

如果检测节点只给出二维中心 $(u,v)$，需要读取对齐深度 $Z$ 并利用相机内参反投影：

$$
X_C=\frac{(u-c_x)Z}{f_x},\qquad
Y_C=\frac{(v-c_y)Z}{f_y},\qquad
Z_C=Z
$$

其中 $(f_x,f_y)$ 是焦距，$(c_x,c_y)$ 是主点。当前系统使用 `yolo_ros` 的 3D 检测结果，因此 `vision_tf_node.py` 接收到的已经是相机坐标系下的三维中心。

深度是定位误差的重要来源。比“中心像素取一次值”更稳定的工程实现是：

1. 对齐深度图和彩色图；
2. 在检测框中心附近取小区域；
3. 去除 0、NaN 和超出工作范围的值；
4. 使用中值而不是均值抑制离群点；
5. 对连续帧做滑动滤波，并限制单帧位移突变。

## 5. Eye-in-Hand 手眼标定

### 5.1 坐标系定义

| 符号 | 坐标系 |
|---|---|
| $\{B\}$ | 机械臂基坐标系 `base_link` |
| $\{E\}$ | 机械臂末端/法兰坐标系 `Link6` |
| $\{C\}$ | 相机坐标系 `camera_link` |
| $\{K\}$ | 标定板坐标系 |

相机固定在机械臂末端，未知量是相机相对末端的固定变换 $X={}^ET_C$。两次采样满足：

$$
A_1XC_1=A_2XC_2
$$

整理得到：

$$
(A_2^{-1}A_1)X=X(C_2C_1^{-1})
$$

即经典模型 $AX=XB$。RealMan 官方工具负责数值求解，本项目的工程工作是采集有效姿态、保存外参并把结果接入 TF 树。

### 5.2 采样原则

- 标定板固定不动且角点清晰；
- 每次同步保存末端位姿与标定板相机位姿；
- 姿态同时覆盖平移和旋转，避免只在小范围平移；
- 标定后用多个已知点检查变换，而不是只看工具提示“成功”；
- 相机支架、末端连接件或安装角度变化后重新标定。

公开外参见 [`config/hand_eye_calibration.yaml`](../config/hand_eye_calibration.yaml)。它仅对应本项目装配，不能直接复制到另一台设备。

## 6. 相机坐标到机械臂基坐标

相机测得目标点 ${}^{C}p_O$ 后，通过 TF 链计算：

$$
{}^{B}p_O={}^BT_E(q)\;{}^ET_C\;{}^{C}p_O
$$

- ${}^BT_E(q)$：由当前关节状态和正运动学实时决定；
- ${}^ET_C$：手眼标定得到的固定外参；
- ${}^{C}p_O$：深度相机测得的目标三维点。

`vision_tf_node.py` 不手写矩阵相乘，而是查询 `camera_link → base_link` 的 TF，再调用 `do_transform_point`。这样可以统一处理变换方向、组合关系和 ROS 时间戳。

输出话题为：

```text
/grasp_target_pose  (geometry_msgs/PoseStamped)
```

当前公开版本把姿态设置为单位四元数，只使用目标位置。这适合验证定位与机械臂主链路，但不是完整的 6D 抓取姿态估计。面对倾斜或不规则物体，需要额外估计抓取方向。

## 7. RM65、灵巧手与 TCP 建模

### 7.1 为什么不能直接把 `Link6` 当抓取点

`Link6` 表示机械臂法兰，实际抓取中心位于手掌前方。如果 IK 只让法兰到达目标，灵巧手会保留固定几何偏差。

因此合并模型增加两段固定关系：

```text
Link6 --rm65_to_lites001_joint--> hand_base_link
hand_base_link --hand_to_tcp_joint--> tcp_link
```

本项目配置快照中，法兰到手掌的安装偏移为 `xyz="0 0 0.006"`、`rpy="0 1.5708 3.1416"`；手掌到 TCP 的示例偏移为 `xyz="-0.18 0 0"`。后者需要结合真实手指接触区域继续实测修正。

SRDF 将规划链定义为：

```xml
<chain base_link="base_link" tip_link="tcp_link"/>
```

URDF、SRDF、IK 请求中的 `ee_link` 必须使用一致名称。

### 7.2 运动学求解

IK 可以理解为：

$$
q^*=\arg\min_q \|f(q)-x_{target}\|
$$

MoveIt 使用 KDL 插件求解，并用实时 `/joint_states` 作为种子。返回结果不能直接按数组位置发送，而是先建立 `joint_name → position` 映射，再按控制器要求的 `joint1...joint6` 顺序排列。

## 8. 抓取控制节点

### 8.1 状态与通信接口

| 类型 | 名称 | 用途 |
|---|---|---|
| Topic | `/grasp_target_pose` | 接收基坐标系目标位姿 |
| Topic | `/joint_states` | 提供当前关节状态和 IK 初值 |
| Service | `/compute_ik` | MoveIt 逆运动学 |
| Action | `/rm_group_controller/follow_joint_trajectory` | 执行 6 关节轨迹 |

### 8.2 默认参数

| 参数 | 默认值 | 设计目的 |
|---|---:|---|
| `approach_height` | 0.10 m | 先移动到目标上方，降低直接碰撞风险 |
| `lift_height` | 0.15 m | 抓取后垂直提起 |
| `trajectory_duration_sec` | 3.0 s | 首轮真机联调采用较慢动作 |
| `ik_timeout_sec` | 1.5 s | 限制 IK 求解时间 |
| `future_wait_timeout_sec` | 2.0 s | 防止 Service/Action 无限等待 |
| `goal_cooldown_sec` | 5.0 s | 避免连续检测触发重复抓取 |
| `goal_position_threshold` | 0.01 m | 1 cm 内目标在冷却期视为重复 |
| `ee_link` | `tcp_link` | 以真实抓取中心而非法兰作为 IK 末端 |

### 8.3 顺序状态机

```text
等待 joint_states
  → 接收目标并加锁
  → 张开手
  → approach: 目标 z + 0.10 m
  → descend: 目标位置
  → 闭合手
  → lift: 目标 z + 0.15 m
  → 释放状态锁
```

抓取流程运行在后台线程中，ROS 节点使用 `MultiThreadedExecutor` 和 `ReentrantCallbackGroup`。这样等待 IK 或轨迹结果时，响应回调仍有线程可以运行，避免“Future 等响应，但执行器被自己占住”的死锁式卡顿。

## 9. 规则抓取策略

| 识别形状 | 策略 | 直观原因 |
|---|---|---|
| 圆形 | 五指包络抓取 | 增大接触范围，对中心小偏差更宽容 |
| 方形 | 拇指与对向手指捏取 | 利用相对平整表面形成稳定夹持 |

这是一种规则驱动的 MVP，优点是可解释、易调试。它没有显式处理物体姿态、摩擦系数、质量或接触力，因此不能等同于通用灵巧抓取算法。后续应把策略输入从“类别”扩展为完整 6D grasp pose、目标尺寸和触觉反馈。

## 10. 动力学与执行安全

机器人动力学形式为：

$$
M(q)\ddot q+C(q,\dot q)\dot q+g(q)+\tau_f=\tau
$$

本项目没有自行实现逆动力学控制，而是由 RM65 底层伺服跟踪 ROS 2 发送的关节轨迹。动力学知识主要体现在：

- 正确设置工具负载、质心和 URDF 惯性参数；
- 用关节限制和较长轨迹时间降低惯性冲击；
- Approach/Descend/Lift 分段执行，便于控制碰撞风险；
- 末端加装 540 g 灵巧手、相机和支架后重新检查负载能力；
- 抓取力不能只依赖位置闭合，精细抓取需要电流、力或触觉闭环。

## 11. 实验如何评价

论文完成了功能性验证：YOLO 在复杂背景、光照变化、角度变化和部分遮挡下比 HSV/轮廓基线稳定；手眼变换能引导机械臂运动到目标附近；系统能够执行识别、接近、闭合和提起链路。同时也记录了抓取失败案例。

目前没有足够数据支持宣称固定抓取成功率。更规范的下一轮实验应记录：

| 层级 | 建议指标 |
|---|---|
| 检测 | Precision、Recall、mAP50、mAP50-95、单帧延迟 |
| 深度 | 静态目标深度均值、标准差、无效像素率 |
| 标定/定位 | 多点位置误差、重复定位误差、不同工作区误差 |
| 运动 | IK 成功率、规划/执行时间、控制器拒绝率 |
| 抓取 | 按类别统计成功率、滑落率、失败原因 |

建议把失败归因拆成：未检测、深度无效、TF 缺失、IK 失败、轨迹拒绝、定位偏差、手型不匹配和抓持滑落。只有这样才能知道优化哪一层最划算。

## 12. 已知限制与下一步

1. 当前只使用三维中心，缺少目标姿态与表面法向；
2. D435i 对反光、弱纹理和物体边缘的深度可能波动；
3. 手眼外参对支架松动敏感；
4. 公开代码默认固定工具朝向，目标倾斜时适应性有限；
5. 灵巧手驱动与类别-手型参数没有在公开仓库中完整复现；
6. 缺少按类别统计的定位误差和抓取成功率。

优先级最高的改进不是立刻换更复杂模型，而是先建立可重复实验记录：保存检测结果、原始深度、TF 变换、IK 错误码、目标关节角和最终成功/失败标签。完整数据链建立后，才能可靠判断问题来自感知、标定、规划还是抓取接触。
